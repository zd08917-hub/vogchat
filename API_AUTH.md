# Документация по улучшенной системе аутентификации

## Обзор

Была реализована гибридная система аутентификации, которая позволяет пользователям:
1. Входить по email-кодам (существующий функционал)
2. Устанавливать пароль для быстрого входа (новый функционал)
3. Выбирать предпочтительный метод входа
4. Сбрасывать пароль через email

## Изменения в базе данных

### Миграция `20260501062445_add_password_fields.js`
Добавлены новые поля в таблицу `users`:
- `has_password` (boolean) - есть ли у пользователя пароль
- `password_set_at` (timestamp) - когда был установлен пароль
- `preferred_login_method` (enum: 'password', 'email_code') - предпочтительный метод входа
- Индекс на поле `has_password` для оптимизации запросов

## Новые API эндпоинты

### 1. Установка/изменение пароля
**POST /api/auth/set-password**

**Параметры:**
```json
{
  "userId": "string (обязательно)",
  "password": "string (обязательно, минимум 8 символов, буквы и цифры)",
  "currentPassword": "string (опционально, требуется при изменении существующего пароля)"
}
```

**Ответ:**
```json
{
  "success": true,
  "message": "Пароль успешно установлен",
  "hasPassword": true,
  "preferredLoginMethod": "password"
}
```

### 2. Вход по паролю
**POST /api/auth/login-password**

**Параметры:**
```json
{
  "email": "string (обязательно)",
  "password": "string (обязательно)"
}
```

**Ответ:**
```json
{
  "success": true,
  "user": {
    "id": "string",
    "email": "string",
    "name": "string",
    "avatarUrl": "string",
    "hasPassword": true,
    "preferredLoginMethod": "password"
  }
}
```

**Ошибки:**
- `400` - У пользователя не установлен пароль (требуется email-код)
- `401` - Неверный пароль
- `404` - Пользователь не найден

### 3. Проверка наличия пароля
**GET /api/auth/has-password**

**Параметры (query):**
- `userId` - ID пользователя

**Ответ:**
```json
{
  "hasPassword": true,
  "preferredLoginMethod": "password"
}
```

### 4. Установка предпочтительного метода входа
**POST /api/auth/preferred-login-method**

**Параметры:**
```json
{
  "userId": "string (обязательно)",
  "method": "string (обязательно, 'password' или 'email_code')"
}
```

**Ответ:**
```json
{
  "success": true,
  "preferredLoginMethod": "password"
}
```

### 5. Запрос сброса пароля
**POST /api/auth/reset-password-request**

**Параметры:**
```json
{
  "email": "string (обязательно)"
}
```

**Ответ:**
```json
{
  "success": true,
  "message": "Код сброса пароля отправлен на email",
  "emailSent": true,
  "userId": "string"
}
```

### 6. Сброс пароля с кодом
**POST /api/auth/reset-password**

**Параметры:**
```json
{
  "userId": "string (обязательно)",
  "code": "string (обязательно, 6-значный код из email)",
  "newPassword": "string (обязательно, минимум 8 символов, буквы и цифры)"
}
```

**Ответ:**
```json
{
  "success": true,
  "message": "Пароль успешно сброшен"
}
```

## Изменения в отправке email

### Файл `server/src/email.js`
Полностью переработан для поддержки:
1. **Реальной отправки email через SMTP** (если настроены креденшиалы)
2. **HTML-шаблонов** для разных целей (регистрация, вход, сброс пароля)
3. **Режима разработки** - логирование кодов в консоль, если SMTP не настроен
4. **Обработки ошибок** с подробными сообщениями

**Настройка SMTP:**
```env
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=your_email@gmail.com
SMTP_PASS=your_app_password
NODE_ENV=production
```

## Вспомогательные функции

### В серверной части (`server/src/index.ts`)
```typescript
// Хэширование пароля
async function hashPassword(password: string): Promise<string>

// Проверка пароля
async function verifyPassword(password: string, hash: string): Promise<boolean>

// Валидация пароля
function validatePassword(password: string): { valid: boolean; errors: string[] }
```

**Требования к паролю:**
- Минимум 8 символов
- Должен содержать хотя бы одну букву
- Должен содержать хотя бы одну цифру

## Обратная совместимость

1. **Существующие пользователи** продолжают работать через email-коды
2. **Новые поля в БД** имеют значения по умолчанию:
   - `has_password`: false
   - `preferred_login_method`: 'email_code'
3. **API эндпоинты** не ломают существующий функционал

## Безопасность

1. **Хранение паролей** - используется bcrypt с salt rounds = 10
2. **Валидация паролей** - проверка сложности на стороне сервера
3. **Защита от брутфорса** - стандартные механизмы bcrypt
4. **Сброс пароля** - через временные коды с ограничением по времени (15 минут)

## Тестирование

### Сценарии тестирования:
1. Регистрация нового пользователя → установка пароля → вход по паролю
2. Вход существующего пользователя по email-коду → установка пароля → вход по паролю
3. Смена пароля с проверкой текущего пароля
4. Сброс пароля через email
5. Переключение между методами входа
